보세전시장에 대한 챗봇을 만드려고 해. 

좋아요. 보세전시장용 챗봇은 일반 안내 챗봇보다 규정 + 운영 + 고객응대를 같이 다뤄야 해서, 처음부터 범위를 잘 나누는 게 중요합니다.

우선 이렇게 정의하면 설계가 쉬워집니다.

1. 챗봇의 역할부터 정하기

보세전시장 챗봇은 보통 아래 4가지를 처리하면 가장 실용적입니다.

방문객 안내
전시장 위치, 운영시간, 입장 절차
주차, 출입 동선, 편의시설
행사 일정, 참가 업체 정보
보세 관련 안내
보세전시장 반입/반출 절차
통관 관련 기본 안내
신고 대상, 필요 서류, 유의사항
“이 물품 전시 가능해?” 같은 기본 질의
참가업체 지원
부스 신청, 반입 일정, 물류 절차
장치/철거 일정
샘플품 반출, 재반출, 폐기 등 안내
담당자 연결
운영자 업무 보조
FAQ 자동응답
민원/문의 분류
상담원 연결
공지사항 자동 안내

즉, 이 챗봇은 단순한 “전시장 안내봇”이 아니라
방문객 응대 + 참가사 지원 + 보세업무 FAQ + 상담 연결까지 묶인 형태가 좋습니다.

2. 먼저 정해야 할 핵심 질문

챗봇을 만들기 전에 아래를 정하면 방향이 잡힙니다.

A. 누구를 대상으로 하나

대상이 섞이면 답변 방식이 달라집니다.

일반 방문객
참가업체
물류/통관 담당자
내부 운영자

가장 좋은 방법은 첫 화면에서 이렇게 나누는 겁니다.

방문객 문의
참가업체 문의
보세/통관 문의
담당자 연결

이렇게만 해도 정확도가 확 올라갑니다.

B. 어디서 쓸 건가

채널에 따라 구현 방식이 달라집니다.

웹사이트 챗봇
카카오톡 채널
전시장 키오스크
사내 운영페이지

처음에는 웹사이트 + 관리자 FAQ 연동으로 시작하는 게 가장 현실적입니다.

C. 어디까지 답하게 할 건가

보세 관련은 민감해서 범위를 정해야 합니다.

추천 범위:

일반 절차 설명: 가능
서류 준비 안내: 가능
규정 근거 문서 안내: 가능
개별 통관 판단/법률 해석: 제한
최종 판단: 담당자 연결

즉,
“안내는 챗봇, 판단은 사람”
구조가 안전합니다.

목표는 단순 FAQ 봇이 아니라, 규정 기반 안내·민원 분류·신청 지원·상담 이관·감사 추적까지 포함한 공공기관형 업무지원 챗봇으로 설계하는 것입니다.

보세전시장은 관세법상 박람회·전람회·견본품 전시회 등의 운영을 위해 외국물품을 장치·전시·사용할 수 있는 보세구역이고, 특허보세구역을 설치·운영하려면 세관장의 특허가 필요합니다. 또 보세전시장 안에서의 “사용”에는 물품의 성질·형상 변경과 전시장 안에서의 소비가 포함됩니다.

1. 서비스 목표

이 챗봇의 1차 목표는 아래 4가지입니다.

민원인 자기해결률 향상
신청 예정자, 행사 주최자, 전시장 운영자, 출품자가 반복적으로 묻는 규정·절차 질문을 24시간 안내합니다. 관세청 FAQ에서도 실제로 보세전시장 관련 문의가 반복되고 있으며, 반입신고·검사·판매용품·증여물품·시식용 식품 같은 실무 질문이 자주 발생합니다.
세관 상담부하 감소
“어디에 신청하나요”, “어떤 서류가 필요한가요”, “판매 가능한가요”, “시식 가능한가요”, “반출은 어떻게 하나요” 같은 1차 민원을 챗봇이 소화합니다. 보세전시장 특허 및 관련 업무는 실제로 세관 내 담당 부서가 맡고 있습니다.
규정 일관성 확보
답변은 모델의 추측이 아니라 관세법·시행령·관세청 고시·공식 FAQ에 근거해 생성합니다.
민원 처리의 디지털 전환
단순 질의응답을 넘어, 상담 예약·문의 접수·관할 세관 안내·증빙 업로드·상담 이력 관리로 연결합니다. 세관별로 보세전시장 특허·관리 담당이 공개되어 있어 실제 민원 연계 구조를 설계할 수 있습니다.

9. 대화 흐름 설계
9-1. 대표 흐름 1: 신청 예정자

사용자: 보세전시장을 신청하려고 해요
→ 챗봇: 전시장 운영자이신가요, 행사 주최자이신가요?
→ 사용자 선택
→ 챗봇: 신청 개요 / 필요 서류 / 관할 세관 안내 / 상담 연결 중 무엇이 필요하신가요?
→ 선택
→ 근거 조문 + 단계형 안내
→ 체크리스트 제공
→ “관할 세관 연결이 필요하시면 문의 접수까지 도와드리겠습니다.”

9-2. 대표 흐름 2: 전시품 사용 문의

사용자: 전시장 안에서 외국물품을 판매해도 되나요?
→ 챗봇: 판매용 전시품인지, 증여용인지, 시연용인지 알려주세요
→ 사용자 입력
→ 챗봇: 관련 규정·FAQ 검색
→ 답변: 원칙 / 조건 / 신고 필요성 / 최종 확인처 안내

보세전시장 내 사용 범위는 시행령 제208조에서 “성질·형상 변경”과 “전시장 안 소비”를 포함한다고 보고 있고, 공식 FAQ는 판매용품·오락용품·증여물품에 대한 실무 안내를 제공합니다.

9-3. 대표 흐름 3: 시식용 식품

사용자: 해외 식품 샘플을 시식용으로 배포해도 되나요?
→ 챗봇: 행사 홍보/시식용인지, 무상 증여인지 확인
→ 챗봇: 관세법 및 식약처 관련 요건 안내
→ 필요시 식약처 문의 연결 문구 제시

관세청 FAQ는 보세전시장 운영자가 박람회 등 행사에서 홍보·시식용으로 사용하는 수입식품에 대해 식약처 제도와 연결된 안내를 제공하고 있습니다.

21. 실제 답변 예시 정책
예시 1

질문: 보세전시장은 뭔가요?
답변:

보세전시장은 박람회·전람회·견본품 전시회 등의 운영을 위해 외국물품을 장치·전시·사용할 수 있는 보세구역입니다. 관세법 제190조에 근거합니다.
예시 2

질문: 보세전시장을 운영하려면 허가가 필요한가요?
답변:

예. 보세전시장은 특허보세구역에 해당하므로 설치·운영하려는 자는 세관장의 특허를 받아야 합니다. 기존 특허의 갱신도 동일합니다.
예시 3

질문: 전시장에서 판매도 가능한가요?
답변:

보세전시장 내 사용할 외국물품은 세관장에게 반입신고하고 검사를 받아야 하며, 공식 FAQ는 판매용품·오락용품·증여물품을 수입신고 수리 후 사용 가능한 유형으로 안내합니다. 다만 실제 판매 방식과 물품 유형에 따라 절차가 달라질 수 있으므로 관할 세관 확인이 필요합니다.
예시 4

사용자 유형과 행사 유형에 따라 제출서류·점검항목을 제시한다.

FR-07 FAQ 추천
현재 질문과 유사한 FAQ 3~5개를 제시한다.

FR-08 로그 및 감사
질문, 검색 근거, 생성 답변, 정책 판정, 이관 여부를 저장한다.

FR-09 관리자 기능
FAQ, 규정 근거, 공지사항, 이관 정책, 실패 질문을 관리한다.

FR-10 버전관리
법령 시행일과 개정 이력을 기준으로 지식 버전을 관리한다. 관세법과 시행령 페이지에는 현재 시행일이 명시되어 있어 버전 기준 데이터로 활용할 수 있습니다.

variant_suffixes = [
    "절차가 어떻게 되나요?",
    "어디에 문의해야 하나요?",
    "미리 준비해야 할 것은 무엇인가요?",
    "주의할 점이 있나요?",
]

expanded = []
for item in faq_items:
    expanded.append(item)
    base_q = item["canonical_question"].rstrip("?")
    for suf in variant_suffixes:
        if len(expanded) >= 300:
            break
        newq = base_q + " " + suf
        expanded.append({
            **item,
            "faq_id": f"FAQ-{faq_id:04d}",
            "canonical_question": newq,
            "answer_short": item["answer_short"],
            "answer_long": item["answer_long"] + " 실제 적용은 행사 구조와 물품 성격을 함께 검토해야 합니다.",
        })
        faq_id += 1
    if len(expanded) >= 300:
        break

expanded = expanded[:300]

    writer.writeheader()
    for row in expanded:
        writer.writerow({k: row[k] for k in writer.fieldnames})

7-2. 테이블 상세
1) intent
CREATE TABLE intent (
    intent_id              VARCHAR(100) PRIMARY KEY,
    intent_name_ko         VARCHAR(200) NOT NULL,
    description            TEXT,
    risk_level             VARCHAR(20) NOT NULL,
    answer_policy          VARCHAR(50) NOT NULL,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW()
);
2) entity_type
CREATE TABLE entity_type (
    entity_type_id         VARCHAR(100) PRIMARY KEY,
    description            TEXT,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW()
);
3) entity_value
CREATE TABLE entity_value (
    entity_value_id        BIGSERIAL PRIMARY KEY,
    entity_type_id         VARCHAR(100) NOT NULL REFERENCES entity_type(entity_type_id),
    value_code             VARCHAR(100) NOT NULL,
    value_label            VARCHAR(200) NOT NULL,
    sort_order             INT NOT NULL DEFAULT 0,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE(entity_type_id, value_code)
);
4) faq
CREATE TABLE faq (
    faq_id                 VARCHAR(50) PRIMARY KEY,
    intent_id              VARCHAR(100) NOT NULL REFERENCES intent(intent_id),
    canonical_question     TEXT NOT NULL,
    answer_short           TEXT NOT NULL,
    answer_long            TEXT NOT NULL,
    risk_level             VARCHAR(20) NOT NULL,
    escalation_rule        TEXT,
    owner_dept             VARCHAR(200),
    status                 VARCHAR(30) NOT NULL DEFAULT 'draft',
    version_no             INT NOT NULL DEFAULT 1,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    last_reviewed_at       DATE,
    created_by             VARCHAR(50),
    updated_by             VARCHAR(50),
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW()
);
5) faq_variant
CREATE TABLE faq_variant (
    faq_variant_id         BIGSERIAL PRIMARY KEY,
    faq_id                 VARCHAR(50) NOT NULL REFERENCES faq(faq_id) ON DELETE CASCADE,
    variant_question       TEXT NOT NULL,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW()
);
6) faq_citation

CREATE TABLE IF NOT EXISTS intent (
    intent_id              VARCHAR(100) PRIMARY KEY,
    intent_name_ko         VARCHAR(200) NOT NULL,
    description            TEXT,
    risk_level             VARCHAR(20) NOT NULL CHECK (risk_level IN ('low', 'medium', 'high', 'critical')),
    answer_policy          VARCHAR(50) NOT NULL,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS entity_type (
    entity_type_id         VARCHAR(100) PRIMARY KEY,
    description            TEXT,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at             TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS entity_value (
    entity_value_id        BIGSERIAL PRIMARY KEY,
    entity_type_id         VARCHAR(100) NOT NULL REFERENCES entity_type(entity_type_id),

    value_label            VARCHAR(200) NOT NULL,
    sort_order             INT NOT NULL DEFAULT 0,
    is_active              BOOLEAN NOT NULL DEFAULT TRUE,
    created_at             TIMESTAMP NOT NULL DEFAULT NOW(),

    UNIQUE(entity_type_id, value_code)
);

CREATE OR REPLACE FUNCTION fn_update_faq_tsv() RETURNS trigger AS $$
BEGIN
    NEW.canonical_question_tsv :=
        to_tsvector('simple', unaccent(COALESCE(NEW.canonical_question, '')));
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION fn_update_faq_variant_tsv() RETURNS trigger AS $$
BEGIN
    NEW.variant_question_tsv :=
        to_tsvector('simple', unaccent(COALESCE(NEW.variant_question, '')));
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION fn_update_rag_document_tsv() RETURNS trigger AS $$
BEGIN
    NEW.content_tsv :=
        to_tsvector('simple', unaccent(COALESCE(NEW.title, '') || ' ' || COALESCE(NEW.content, '')));
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION fn_update_rag_chunk_tsv() RETURNS trigger AS $$
BEGIN
    NEW.chunk_text_tsv :=
        to_tsvector('simple', unaccent(COALESCE(NEW.chunk_text, '')));
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_update_faq_tsv ON faq;
CREATE TRIGGER trg_update_faq_tsv
BEFORE INSERT OR UPDATE OF canonical_question
ON faq
FOR EACH ROW
EXECUTE FUNCTION fn_update_faq_tsv();

실무적으로는 아래 8단계로 진행하면 됩니다.

1. 기관 내부에서 먼저 사업 범위를 확정

가장 먼저 정해야 할 것은 “이 챗봇이 무엇까지 답하고, 어디서 사람에게 넘길지”입니다. 지금 설계한 보세전시장 챗봇은 규정 안내, 신청 전 체크리스트, FAQ, 상담 이관까지는 자동화할 수 있지만, 개별 물품의 적법성이나 과세 확정 같은 최종 판단은 담당 세관으로 넘기는 구조가 맞습니다. 이런 범위 정의를 먼저 해야 이후에 개인정보, 민원 처리, 상담 운영 책임도 정리됩니다. 행정안전부의 웹사이트 구축·운영 가이드는 웹사이트를 구축부터 운영·폐기까지 전 주기로 관리하라고 안내하고 있습니다.

2. 홈페이지 연계 방식 결정

다음으로는 기관 홈페이지에 챗봇을 어떤 방식으로 붙일지 결정해야 합니다. 보통은 세 가지입니다.
첫째, 홈페이지 하단의 플로팅 버튼으로 여는 웹 위젯.
둘째, 별도 상담 페이지로 이동하는 전체화면 챗봇.
셋째, 기존 민원/FAQ 페이지 안에 넣는 임베디드형입니다.
공공기관은 접근성과 유지보수를 생각하면 대개 웹 위젯 + 별도 전체화면 페이지를 같이 두는 편이 좋습니다. 이렇게 하면 메인 사이트 UX를 해치지 않으면서도, 긴 상담이나 파일 첨부가 필요한 경우 별도 화면으로 확장할 수 있습니다. 웹사이트 품질관리 가이드가 UI/UX와 접근성 기준을 함께 반영하도록 안내하는 것도 이 단계에서 중요합니다.

3. 홈페이지 CMS/백엔드와 챗봇 API 연결 설계

지금 만든 openapi.yaml과 schema.sql을 기준으로, 기관 홈페이지와 챗봇 서버 사이의 연결 구조를 정해야 합니다. 일반적으로 홈페이지는 프런트 역할만 하고, 챗봇 엔진은 별도 API 서버에서 처리합니다. 흐름은 홈페이지 UI → /chat/sessions → /chat/messages → 답변 렌더링 → 필요시 /chat/handoffs 구조가 가장 단순합니다. 관리자 화면은 홈페이지와 분리된 내부망 또는 관리자 도메인에서 운영하는 편이 안전합니다. 공공기관 웹사이트 가이드도 구축 시 표준기술, 최적화, 운영관리 전반을 포함해 설계하라고 하고 있습니다.

4. 개인정보 및 법적 고지 정리

홈페이지에 챗봇을 붙이면, 질문 자체는 익명으로 받아도 상담 이관 시 이름·전화번호·이메일을 수집하게 됩니다. 그래서 최소한 다음은 같이 준비해야 합니다.

개인정보 처리방침 반영
챗봇 전용 수집·이용 안내 문구
보유기간/파기 기준
상담 이관 동의 체크
자동응답 한계와 최종 판단 주체 고지
행정안전부 가이드가 개인정보보호 관련 고시와 국가정보보안 기본지침 등의 반영을 언급하고 있고, KISA도 최근 공공 상담 서비스 개편에서 채팅·챗봇을 도입하면서 접근성과 원스톱 상담 구조를 강조했습니다.
5. 접근성·품질 기준에 맞춰 프런트 구현

공공기관 홈페이지에 붙이는 순간 가장 자주 놓치는 부분이 접근성입니다. 챗봇 버튼, 대화창, 추천질문, 상담 이관 폼이 모두 키보드만으로 조작 가능해야 하고, 스크린리더가 읽을 수 있어야 하며, 명도 대비와 포커스 이동도 맞춰야 합니다. 2025년 수정된 전자정부 웹사이트 품질관리 가이드는 접근성 기준을 KWCAG 2.2로 바꿨고, 웹 편의성 기준도 디지털 정부서비스 UI/UX 가이드라인 중심으로 재편했습니다. 따라서 챗봇 UI도 일반 팝업이 아니라 “공공 웹 컴포넌트”처럼 구현해야 합니다.

6. 지식데이터 적재와 운영 검수

그다음은 지금 만든 패키지의 intents.json, entities.json, faq_seed_300.csv, rag_documents_seed_50.jsonl를 실제 운영 DB와 벡터 인덱스에 넣고, 실무부서 검수를 거쳐야 합니다. 특히 보세전시장은 법령, 시행령, 고시, FAQ, 세관 담당부서 정보가 함께 맞물리므로, 초기 오픈 전에는 반드시 담당 부서가 “자동응답 가능 / 사람 이관 필요” 경계를 확인해야 합니다. 공공기관 챗봇은 기술보다 답변 책임 범위가 더 중요합니다. 행정안전부 가이드가 관련 법령 개정사항을 반영해 운영하라고 하는 이유도 여기에 있습니다.

7. 시범 운영 후 정식 오픈

바로 홈페이지 메인에 공개하지 말고, 먼저 시범 운영을 권합니다. 가장 좋은 방식은

내부 직원 테스트
